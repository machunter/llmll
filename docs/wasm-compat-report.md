## GHC WASM Backend: Technology Overview

The [GHC WASM backend](https://gitlab.haskell.org/ghc/ghc/-/wikis/javascript-backend) generates `.wasm` modules from standard GHC Haskell. As of GHC 9.6+:

- **Invocation:** `wasm32-wasi-ghc --make -isrc src/Lib.hs`
- **Output:** `.wasm` module + WASI import stubs
- **Prelude coverage:** Full `base` library compiles to WASM; `System.IO`, `Data.List`, `Data.Map.Strict` all compile cleanly

Install: `ghcup install ghc --target=wasm32-wasi 9.6.6`

### ⚠️ Items requiring attention before v0.4 WASM deployment

| Item | Status | Mitigation |
|------|--------|-----------|
| `System.Random` (for `random_int`) | Needs WASI entropy polyfill | `wasi_random_get` syscall; replace `return 42` stub |
| `foreign import ccall` FFI stubs (`c.*` imports) | Must use WASI or wasm-host imports | Regenerate stub template for `foreign import "wasm"` |
| `warp` HTTP server (`haskell.warp`) | Not WASM-compatible | Replace with `servant-wasm` or WASI httpx bindings in v0.4 |
| `IORef`/`STRef` | Compiles but thread model differs | Single-threaded WASI is fine; avoid `Control.Concurrent` |
| Stack overflow in deep recursion | WASM default stack is smaller | Add `{-# OPTIONS_GHC -O2 #-}` to generated files; use tail-recursive patterns |

### No structural blockers

None of the patterns generated by the v0.1.2 Haskell backend are fundamentally incompatible with WASM. All LLMLL programs that use only the `§13` stdlib and pure Haskell logic can be compiled to WASM today with `wasm32-wasi-ghc` without changes to the emitter.

## Validation Command (when GHC WASM backend is installed)

```bash
# Install WASM GHC (one-time)
ghcup install ghc --target=wasm32-wasi 9.6.6
ghcup set ghc wasm32-wasi-9.6.6

# Compile hangman to WASM
llmll build examples/hangman_complete.llmll --output /tmp/hangman-wasm --wasm
cd /tmp/hangman-wasm && wasm32-wasi-ghc --make -isrc src/Lib.hs -o hangman.wasm

# Run under WASI runtime
wasmtime hangman.wasm
```
