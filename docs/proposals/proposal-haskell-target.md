# Proposal: Move LLMLL Code Generation Target from Rust to Haskell

> **Status:** Discussion Draft  
> **Authors:** PhD Research Team  
> **Related:** `LLMLL.md §14` · `docs/project-management/roadmap.md`

---

## 1. Problem Statement

LLMLL's compiler is written in Haskell, yet its code generation target is Rust. This creates a **semantic impedance mismatch**: LLMLL's core model — pure functions, ADTs, immutability, algebraic effects, liquid type contracts — maps directly onto Haskell's native semantics, yet codegen translates these constructs into an ownership-based, mutation-capable language that was not designed for them.

The primary justification for Rust was **WASM-first sandboxing**. This proposal argues that:

1. WASM is not a critical constraint at the current research stage.
2. Practical sandboxing can be achieved through Docker, OS primitives, and Safe Haskell.
3. A Haskell target eliminates entire roadmap work-items that are already solved in the Haskell ecosystem.
4. A clear, incremental path to WASM exists for eventual production hardening.

---

## 2. What Changes

| Aspect | Current (Rust target) | Proposed (Haskell target) |
|--------|----------------------|--------------------------|
| Compiler language | Haskell | Haskell (unchanged) |
| Generated code | Rust (`src/lib.rs`, `Cargo.toml`) | Haskell (`.hs` + `package.yaml`) |
| Runtime host | Rust (Wasmtime) | GHC runtime + OS sandbox |
| FFI stubs | `src/ffi/*.rs` | `src/ffi/*.hs` (GHC `foreign import ccall`) |
| Standard library | Thin; much requires FFI | Hackage — rich native ecosystem |
| Liquid types (v0.2) | Build from scratch | **LiquidHaskell** — exists today |
| Property tests | QuickCheck (compiler only) | QuickCheck — same tool for compiler and generated code |
| Sandboxing | WASM VM | Docker + `seccomp-bpf` + `-XSafe` |
| Path to WASM | Done (current) | GHC native WASM backend (v0.4+) |

## 3. Semantic Alignment Gains

Every LLMLL concept has a native Haskell counterpart the codegen can emit directly:

| LLMLL | Generated Haskell |
|-------|-------------------|
| `def-logic f [x: int] body` | `f :: Int -> <inferred>; f x = body` |
| `(type T (| A t) (| B s))` | `data T = A t \| B s` |
| `Result[t,e]` | `Either e t` |
| `Command` / effect model | `Eff '[FileSystem, Http] a` (algebraic effects) |
| `def-interface I [m fn-type]` | `class I a where m :: fn-type` |
| `pre`/`post` contracts | LiquidHaskell refinement annotations |
| `check` / `for-all` | QuickCheck `property` / `forAll` |

The generated Haskell is idiomatic, directly readable, and shares the compiler's own type universe. No borrow checker contortions required.

## 4. FFI System Redesign

### 4.1 What disappears

The vast majority of the current FFI standard library has direct Haskell equivalents requiring **no FFI at all**:

| Rust FFI (current) | Native Haskell replacement |
|--------------------|---------------------------|
| `rust.serde_json` | `Data.Aeson` |
| `rust.clap` | `Options.Applicative` |
| `rust.atomic_fs` | `System.Posix.Files` + `atomicWriteFile` |
| `rust.env` | `System.Environment` |
| `rust.timer` | `Control.Concurrent.threadDelay` |
| `rust.http_server` | `Network.Wai` / `warp` |

These become regular GHC `import` statements — no stub generation, no `todo!()`, no `Cargo.toml` editing.

### 4.2 Revised Three-Tier FFI

| Tier | Namespace | Mechanism | Use case |
|------|-----------|-----------|----------|
| **1 — Hackage** | `haskell.*` | Regular GHC `import`; no stub generated | Anything on Hackage (20,000+ packages) |
| **2 — C libraries** | `c.*` | GHC `foreign import ccall`; compiler generates `.hs` stub | System libraries, SQLite, libsodium |
| **3 — Python ecosystem** | `python.*` | `inline-python` / `cpython` bridge | NumPy, PyTorch, Pandas |

**Tier 1 example — zero developer action:**
```lisp
;; Resolves to: import Data.Aeson — no stub generated
(import haskell.aeson (interface [
  [json-decode (fn [s: string] -> Result[TodoList, string])]
  [json-encode (fn [td: TodoList] -> string)]
]))
```

**Tier 2 example — C FFI stub generated:**
```lisp
;; Generates src/ffi/Libsodium.hs with: foreign import ccall "crypto_sign" ...
(import c.libsodium (interface [
  [crypto-sign (fn [msg: bytes[64] key: bytes[32]] -> bytes[96])]
]))
```

## 5. Sandboxing Strategy (Without WASM)

A **layered sandboxing model** appropriate for a research language:

### Layer 1 — Source-Level: `-XSafe` Haskell
GHC's Safe Haskell pragma prevents any generated module from importing unsafe primitives (`System.IO`, `Foreign.*`, etc.). A generated file compiled with `-XSafe` **literally cannot perform IO** through any means outside the declared capability model — enforced by the type system, not convention.

```haskell
{-# LANGUAGE Safe #-}
-- Generated by llmll build. This module cannot perform IO.
-- All effects are described as Command values, executed by the host.
module MyService where
...
```

### Layer 2 — Process-Level: `seccomp-bpf`
The host runtime launches the generated program with a `seccomp-bpf` filter (Linux) or `Sandbox.framework` (macOS) that whitelists only the system calls matching declared capabilities. Violations terminate the process with `SIGSYS`.

### Layer 3 — Container-Level: Docker
For production deployment, each LLMLL program runs in its own Docker container with:
- Minimal base image (GHC runtime only)
- Read-only filesystem; writable volumes only for declared `(capability write PATH)` paths
- Network policies enforcing declared `(capability post URL)` constraints
- Resource limits (`--memory`, `--cpus`)

```
LLMLL source
     │  llmll build
     ▼
Generated .hs (compiled with -XSafe)
     │  GHC
     ▼
Native binary
     │
     ▼
Docker container
├── seccomp-bpf (syscall whitelist per capabilities)
├── Read-only filesystem (except declared paths)
├── Network policy (declared URLs only)
└── LLMLL host runtime (interprets Commands, enforces capability list)
```

## 6. Path to WASM (Long-Term)

Moving to Haskell does **not** abandon WASM. GHC's native WASM backend (`--target=wasm32-wasi`) reached production quality in GHC 9.10. The path is incremental:

| Phase | Action | Deliverable |
|-------|--------|------------|
| **v0.1.2 (now)** | Switch codegen to Haskell; Docker sandboxing | Fully functional Docker-deployed programs |
| **v0.2** | Validate generated `.hs` compiles with GHC WASM backend | Confirmed WASM-compatible code patterns |
| **v0.3** | `llmll build --target wasm` as opt-in flag | Generated `.wasm` binary |
| **v0.4** | WASM VM replaces Docker as primary sandbox | Full WASM-first production deployment |

Since `-XSafe` already prevents the unsafe IO patterns that break WASM compilation, the generated code is structurally WASM-compatible from day one.

## 8. Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| GHC WASM backend maturity | Docker used until v0.4; no dependency on WASM until then |
| Python FFI bridge fragility | Limit `python.*` to AI/ML use cases; mark as "experimental tier" |
| LiquidHaskell annotation syntax differs from LLMLL spec | Codegen translates `(where ...)` constraints to `{v: t \| predicate}`; spec unchanged |

---

## 9. Recommendation

Adopt the Haskell code generation target starting at **v0.1.2**. The migration:

1. **Low risk** — codegen is one module (`Codegen.hs`); the rest of the compiler is unchanged
2. **High leverage** — eliminates entire roadmap items (liquid type groundwork, runtime scaffolding)
3. **Reversible** — JSON-AST schema is target-agnostic; switching back to Rust codegen is a `Codegen.hs` rewrite
4. **Aligned with research goals** — semantic clarity and verifiability matter more than deployment optimization at this stage
